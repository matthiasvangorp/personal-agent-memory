## Where we are

**Phase 1 v1 + dashboard tier 1 & 2 + Project subsystem + Phase 2 self-gen
all shipped.** All core infrastructure is in production: Telegram bot
(long-poll), Whisper voice-in, scheduled heartbeat with sleep-gap detection,
Apple Health ingest via HAE webhook, the approval gate, the LogWatcher
self-healing loop, the daily HealthDiary check-in, the Livewire dashboard
at `https://agent.test/dashboard`, **and `MetaCreateSkill` which lets the
agent generate brand-new skills on demand**.

The `Project` subsystem (`app/Project/`, `app/Models/Project.php`,
`ProjectLauncher` skill, migration `2026_04_10_200000_create_projects_tables`)
is our local take on OpenClaw-style agentic work: long-running goal-oriented
projects with their own workspace, heartbeat cadence, and tool loop. One
project = one goal + one workspace dir under `storage/agent/projects/<slug>/`
+ one cron cadence. Each tick is a fresh `AnonymousAgent` session that reads
`HEARTBEAT.md` (source of truth across ticks), does one piece of work, writes
a journal entry, and rewrites `HEARTBEAT.md` for next tick. Tools available
inside a project tick: `WebFetch`, `WorkspaceFile`, `AskUser` (blocks the
project until user replies via Telegram). This replaces the "main agent plans
everything in one megaprompt" anti-pattern with stateless tick loops +
workspace-as-memory, same shape as OpenClaw but stripped down.

## Two agent modes â€” Skills vs Projects

- **Skills** (`app/Skills/`) = short, scoped capabilities the main `PersonalAgent`
  can call as tools. Either on-demand (`SendReminder`, `MemoryEditor`) or
  scheduled via `HasHeartbeat` (`WeatherBriefing`, `LogWatcher`). State lives
  in `storage/agent/skills/<slug>/` JSON or SQLite tables. These are small.
- **Projects** (`app/Project/`) = long-running OpenClaw-style agent loops, each
  with its own goal, workspace, heartbeat cadence and tool loop. State lives
  in `storage/agent/projects/<slug>/` as markdown (workspace-as-memory). The
  main agent spawns a project via `ProjectLauncher`; after that, the project
  ticks on its own cadence and can reach back to the user via `AskUser`.
- Rule of thumb: if the work is a one-shot capability or a recurring watcher,
  it's a Skill. If it's a goal that needs to be pursued over hours/days/weeks
  with its own memory and plan, it's a Project.

## Phase 2 self-gen â€” what landed today (2026-04-11)

End-to-end first run:

1. Matthias typed an `RssDigest` spec in Telegram.
2. Main agent recognized the intent, called `meta:create_skill`.
3. Claude (via the `meta-create-skill` ModelRouter call site, Sonnet 4.6)

   `WeatherBriefing`, `MemoryEditor` as full-source few-shot.
4. `validateSource` accepted it: tokenizer-based forbidden-token scan,
   structural checks (namespace, extends, methods, required schema field,
   schedule-name-before-overlap), `php -l`, size caps.
5. Drafted approval row + Telegram inline button with the full generated source.
6. Matthias tapped Approve.
7. `applyPhase` re-validated, wrote `app/Skills/RssDigest.php`, ran
   `composer dump-autoload`, committed + pushed (commit `fc87978`) via the
   `agent_deploy` key, fired deferred `horizon:terminate`.
8. Container restart, `SkillServiceProvider::discoverSkills()` rediscovered,
   registry now has 11 skills with `RssDigest` registered as `HasHeartbeat`.
9. First real `RssDigest` tick fires automatically tomorrow at 08:00 Budapest.

Two iterations were needed before approval â€” both fixes were in the
`MetaCreateSkill` prompt template itself, not in Claude's output:

- **Iteration 1** generated a working file but with `->name('heartbeat:daily_0800')`
  (cadence slug, not skill slug â€” would collide on a second daily_0800 skill)
  and a `\ud83d\uddc3` emoji escape in a single-quoted PHP string (renders as
  literal `\uâ€¦` text in Telegram). Fix: `cadenceHint` now takes the skill name
  and emits `heartbeat:<snake_skill_name>`; UTF-8 guidance simplified to
  "embed the literal character, no escapes".
- **Iteration 2** broke the JSON envelope itself because the prompt told Claude
  to use `"\u{NNNN}"` PHP escapes â€” valid PHP, invalid JSON. Fix: drop the
  escape advice entirely, plus log the raw LLM response on parse failure so
  the next failure isn't black-boxed.

Other things shipped this session:

- `App\Services\Git\GitPusher` extracted from three inlined copies of the same
  git harness (LogWatcher, MetaCreateSkill, MemoryRepo). Constants
  `KEY_AGENT` / `KEY_MEMORY` for the deploy key paths. Single source of truth.
- `ProjectWorkspace::write()` now auto-commits + pushes every workspace
  mutation to the memory repo via `MemoryRepo::commitAndPushPath`. Project
  workspaces (`storage/agent/projects/<slug>/`) are now part of the memory

## How to resume

A fresh session should be able to pick up from this file alone. Start by
reading USER.md, AGENTS.md, this STATUS.md, and the skills under
`memory/skills/`. The 11 skills in `app/Skills/` are the canonical examples
for any new Skill work; `app/Project/ProjectAgent.php` + `ProjectLauncher.php`
are the canonical example for any new Project work; `MetaCreateSkill.php` is
the canonical example for anything that needs a two-phase approval flow with
full validation. The README in the app root has the daily commands.
